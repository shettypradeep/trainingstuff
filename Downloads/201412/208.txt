And again that is indicative of an incorrect level of abstraction in the
plugin.

The reality is that the tycho-pack200 plugin should probably have a
normalize-sign-pack goal, or better yet there should be a pack200-jar
packaging type that creates the packed jar and supports signing of them, ie
replicating the same goals as the jar plugin but doing the extra work
required to ensure this is performed correctly

The mistake, if you like, is trying to reuse the jar plugin's code at the
pom level and not integrate at a lower level.

There are benefits to integrating at a lower level, i.e. It should be
possible to achieve faster performance for users at the same time by
avoiding intermediary files and writing the packed jar direct from source
binary files, so users get a benefit too... And the jar signing code being
refactored into a shared component to allow for reuse would allow for
things like (self-exec-)war:sign to allow signing self-executable war files
such as jenkins.war without hacking to the same level
